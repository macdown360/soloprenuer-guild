# ソロプレナー・ギルド プロダクトガイドライン

## プロダクトの核

ソロプレナー・ギルドは、一人起業家、フリーランス、個人開発者、副業家が集まるオンライン上の冒険者ギルドです。

案件マッチングサイトではなく、ソロプレナー同士の助け合いと成長を促進するコミュニティプラットフォームとして設計します。

## MVPで最優先する循環

1. クエストを発行する
2. 他の冒険者が応募または完了報告を送る
3. 発行者が提出内容を承認する
4. Gold報酬とTrustが達成者へ反映される
5. 評価に応じた追加Trustが積み上がる

この「クエスト投稿 → クエスト達成 → 信頼獲得」の循環を最重要指標にします。

## 表現ルール（内部仕様と外部向けの分離）

- 外部向け（ユーザーに見せる画面）では、運営都合の語を出さない
- 外部向けで避ける語: MVP、MAU、KPI、異常検知、連打制限、評価操作対策ロジック
- 外部向けでは、体験中心の語に置き換える
  - 例: MVP目標値 → 今月のギルド活動
  - 例: 異常検知 → 不審な評価は運営確認の対象
- 内部仕様（設計書、運用手順、分析基盤）では、数値指標や検知ロジックを明記する
- 画面文言は「ユーザーの行動」と「得られる価値」を先に書き、実装事情は書かない

## 実装する機能

### 会員登録

- 冒険者名
- メールアドレス
- パスワード
- 任意プロフィール: 肩書き、プロフィール、得意な領域、関心テーマ

### マイページ

- 冒険者ランク
- Gold残高
- Trust
- 完了クエスト数
- 発行クエスト数
- プロフィール編集
- 承認待ち/承認済みの参加クエスト表示
- 発行中/クローズ済みクエスト管理
- おすすめクエスト表示

### クエスト

- 作成: タイトル、カテゴリ、詳細説明、参考URL、スクショ画面、種別、募集人数、報酬Gold、タグ、締切
- 種別: 応募型（recruiting）/完了報告型（report）
- 一覧: タイトル、発行者、報酬、締切、進捗（提出数/募集人数）
- 検索: カテゴリ、キーワード、タブ（新着/種別/クローズ）
- 詳細: 内容、発行者情報、進め方、応募または完了報告
- 完了: 発行者承認時に報酬GoldとTrustを達成者へ付与

### 信頼システム

- クエスト完了で +5 Trust
- 5段階評価で追加Trustを付与
- Trustは譲渡不可、売買不可、換金不可

評価ごとの追加Trustは以下で統一します。

- ★★★★★: +10
- ★★★★: +8
- ★★★: +5
- ★★: +2
- ★: +0

## ポイント設計

### Gold

依頼を出すための通貨です。

- 新規登録時に100G付与
- クエスト発行時に報酬分を確保
- 達成承認後、達成者が報酬を100%受け取る
- クエスト削除や編集で未消化エスクローを返却/再計算

### Trust

信用スコアです。

- ランク決定に使う
- 冒険者評価として表示する
- 蓄積型で、通常の依頼発行では消費しない

MVPの最重要ルールは「Goldは増減する、Trustは蓄積する」です。

## ランク

- 見習い冒険者: 0-99
- 駆け出し冒険者: 100-299
- 一人前冒険者: 300-699
- 熟練冒険者: 700-1499
- 上級冒険者: 1500-2999
- マスター冒険者: 3000-4999
- 伝説の冒険者: 5000以上

ランク判定ルールは段階的に運用します。

- MVP: Trustのみでランク判定
- 将来拡張: Trustを主軸に、完了クエスト数と継続利用日数を加点要素として追加

## 運用参照: 実装準拠の算出ルール

この章は、現在の実装と同じルールを運用時に参照するための固定仕様です。
実装ソースは app.js を正とし、この章は変更時に必ず同時更新します。

### 1) ランク判定（Trustのみ）

現在の判定関数は `getRank(trust)` で、閾値は以下です。

- 見習い冒険者: 0-99
- 駆け出し冒険者: 100-299
- 一人前冒険者: 300-699
- 熟練冒険者: 700-1499
- 上級冒険者: 1500-2999
- マスター冒険者: 3000-4999
- 伝説の冒険者: 5000以上

### 2) Gold / Trust の加算・減算イベント

現在の mypage 実装で発生する値変動は以下です。

1. クエスト発行（questForm 送信）
- 条件: 発行報酬 `reward × 募集人数 capacity` が現在Gold以下
- 変動: `Gold -= reward × capacity`, `発行クエスト数 += 1`
- 備考: Trustは変動なし

2. クエスト承認（発行者操作）
- 変動: `Gold += reward`
- 変動: `Trust += (5 + 評価ボーナス)`
- 評価ボーナス: ★5=+10, ★4=+8, ★3=+5, ★2=+2, ★1=+0
- 変動: `完了クエスト数 += 1`

補足:
- Supabase接続時は承認で達成者へ実際に加算
- ローカルデモ時は承認加算は表示のみ（数値への直接加算はしない）

3. デイリーミッション（各ミッション初回達成時のみ）
- ミッション1: `Trust +5`
- ミッション2: `Trust +8, Gold +3`
- ミッション3: `Trust +3`
- 全3件達成ボーナス: `Trust +5`
- 備考: 同一ミッションの再達成加算はなし（state.missions[index] が真なら処理しない）

4. 週次チャレンジ
- 進捗加算トリガー: 承認実行時に `weeklyProgress += 1`（上限3）
- 達成条件: `weeklyProgress >= 3`
- 達成時報酬: `Gold +30, Trust +25`
- 達成後: `weeklyProgress = 0` にリセット
- 備考: 週次カードはローカルデモ時のみ表示

### 3) 称号（現状仕様）

称号表示は mypage の実績セクションで生成します。
一部は条件に応じて獲得表示、未達成はロック表示です。

主な条件文言（UI表示値）

- 初陣: 初クエスト達成
- 助っ人: 10件達成
- 常連冒険者: 7日継続
- 壁打ち名人: 相談回答5件
- 五つ星の協力者: 高評価の実績
- 依頼上手: クエスト発行5件
- ベテラン: 50件達成で獲得

### 4) 運用時チェックポイント

数値事故防止のため、運用前後で以下を確認します。

1. `app.js` の閾値・加算値と本章の数値が一致している
2. 評価プルダウンの表示値と `reviewForm` の加算値が一致している
3. ミッション報酬と週次報酬の重複付与条件が意図通りである
4. 称号条件を動的化する場合は、先に本章へ算出式を追記する

## MVPで実装しないもの

- ユーザー作成ギルド
- クラン機能
- パーティ機能
- 通話機能
- 決済機能
- NFT
- 独自トークン
- AIエージェント

## 追加の実装済み要素

- 応募型提出に対する発行者/応募者チャット
- 参加者向け「承認待ち」「承認済み」一覧
- プロフィール適合スコアによるおすすめクエスト
- クローズ済みクエストで提出履歴を確認
- 入力内容表示時のHTMLエスケープを標準化

## デザイン方針

- ゲーム感はあるが子供っぽくしない
- 中世ファンタジーの冒険者ギルド感と、SaaSの使いやすさを両立する
- 配色はダークネイビー、ゴールド、シルバーを中心にする
- カード、フォーム、一覧、検索、詳細の操作は業務ツールとして分かりやすくする
